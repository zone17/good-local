# Security Review — 001-upper-delaware-passport

Defensive review of the diff `main...001-upper-delaware-passport`. Findings below
are stated neutrally as missing/weak protections, with file:line evidence.

## [P1] No rate limiting on OTP send or OTP verification (brute-force window)
- **File**: supabase/functions/claim-passport/index.ts:73-83 (send), supabase/migrations/0006_rpc_checkin.sql:495-508 (verify)
- **What**: Neither the OTP send path nor the `claim_passport` verify path enforces any rate limit or per-phone attempt counter. The OTP is a 6-digit numeric code (otp-core.ts:8-10) with a 10-minute validity window (index.ts:76). `claim_passport` selects the most-recent matching `(phone, code)` row and raises `OTP_INVALID` on mismatch, but does not record failed attempts, does not lock the phone after N failures, and does not expire prior unconsumed codes when a new one is minted. The `otp_codes` table has no attempt/lockout columns (0006_rpc_checkin.sql:20-27). This leaves a brute-force window over the 10^6 code space and no cap on how many codes can be requested per phone (SMS-cost / inbox-flood exposure on the send path).
- **Fix**: Add a per-phone rate limit on the send path (e.g. max N sends per rolling window, enforced via a count over `otp_codes` by `phone, created_at`). Add a failed-attempt counter / lockout to `claim_passport` (e.g. an `attempts` column on `otp_codes`, increment on mismatch, raise a lockout error past a threshold). Invalidate prior unconsumed codes for the phone when a new one is minted so only one live code exists at a time.

## [P1] Stripe webhook secret falls back to a hardcoded default ("whsec_local_test")
- **File**: supabase/functions/stripe-webhook/index.ts:23
- **What**: `const secret = Deno.env.get("STRIPE_WEBHOOK_SECRET") ?? "whsec_local_test"`. If `STRIPE_WEBHOOK_SECRET` is unset/misconfigured in any deployment, signature verification runs against a known, source-committed secret rather than failing closed. Because `verifyStripeSignature` then validates HMACs computed with that public value, an unauthenticated caller who knows the default could craft a valid `stripe-signature` header for an arbitrary event body and drive the single-writer subscription/billing-status mutations in stripe-webhook-core.ts (activate/suspend/close businesses, write subscription rows). This is the same fail-open shape the project already fixed for dev OTP/mail (docs/solutions/fail-closed-dev-affordances.md): a secret must not be inferable from unset config.
- **Fix**: Treat an unset `STRIPE_WEBHOOK_SECRET` as a hard failure in non-local environments. Require the env var to be present (throw / return 500 if absent) rather than defaulting to a committed literal, mirroring the `?? "production"` fail-closed pattern. At minimum, gate the `whsec_local_test` default behind the same explicit ENVIRONMENT=development/local check used for dev OTP.

## [P1] Public views are non-security_invoker → RLS bypass on stamps/patrons
- **File**: supabase/migrations/0003_views.sql:12-153 (e.g. valid_stamps:12, owner_visit_pattern_14d:149, gate_metrics_all:137)
- **What**: Every view in 0003 is created in the `public` schema (exposed via PostgREST — config.toml:13 `schemas = ["public"]`) without `security_invoker = true`. A non-security_invoker view executes with the VIEW OWNER's privileges and does NOT apply RLS on its underlying tables (`stamps`, `patrons`, etc.). No migration sets `security_invoker`, and no migration revokes SELECT on these views from `anon`/`authenticated` (grep across migrations returns none). Supabase's default privileges grant app roles SELECT on public views. The comment at 0003_views.sql:148 asserts `owner_visit_pattern_14d` is "RLS-scoped at query time via stamps policies" — that holds only under security_invoker, which is not set. As written, an `authenticated` caller can `select * from valid_stamps` (and the gate_* / owner_visit_pattern_14d views) directly through the REST API and read every patron's stamps across every business, bypassing the Art. V boundary that the base-table policies in 0002_rls.sql enforce. The rls-privacy.test.ts suite tests only base tables, so this path is uncovered.
- **Fix**: Add `with (security_invoker = true)` to every view in 0003 (PG15+) so underlying-table RLS applies to the querying role, AND/OR `revoke select on <each view> from anon, authenticated` and surface their data only through the existing SECURITY DEFINER RPCs (gate views are already admin-gated via RPC; owner_visit_pattern_14d/valid_stamps are consumed inside definer functions). Add a regression test asserting a non-owner `authenticated` client gets 0 rows / permission denied selecting `valid_stamps` and `owner_visit_pattern_14d` directly.

## [P2] OTP consumption is check-then-update (TOCTOU); OTP not bound to caller at mint
- **File**: supabase/migrations/0006_rpc_checkin.sql:495-508
- **What**: Two issues in the verify path. (1) TOCTOU: `claim_passport` SELECTs the most-recent `(phone, code)` row and checks `consumed_at is null` (502), then separately `update otp_codes set consumed_at = now()` (508), with no `for update` lock on the selected row. Two concurrent calls presenting the same still-valid code can both pass the unconsumed check before either marks it consumed, so a single OTP can be redeemed more than once. (2) The OTP is minted purely against a phone (claim-passport/index.ts:78-80) with no binding to the requesting caller/auth.uid(); the verify path matches on `(phone, code)` only. Combined with the absence of rate limiting (finding 1), possession of a valid code for a phone is sufficient to bind that phone onto an arbitrary auth account. The phone uniqueness constraint (0001_schema.sql:120) limits the data-corruption blast radius, but does not bind the claim to the party that actually controls the phone beyond the code itself.
- **Fix**: Make consumption atomic — `select ... for update` on the otp row (or do a conditional `update otp_codes set consumed_at = now() where id = v_otp.id and consumed_at is null returning id`, treating zero rows updated as OTP_INVALID). This both closes the TOCTOU and enforces single-use under concurrency. Combined with the lockout in finding 1, this hardens the phone→account binding.

## [P2] Default PUBLIC EXECUTE on functions is never revoked; gate_metrics_evaluated() has no internal authz
- **File**: supabase/migrations/0011_rpc_admin.sql:362-401 (gate_metrics_evaluated); pattern affects all RPC migrations
- **What**: The migrations grant EXECUTE selectively (e.g. 0005:270-274, 0007:236-237, 0009:191-193) but never run `ALTER DEFAULT PRIVILEGES ... REVOKE EXECUTE ON FUNCTIONS FROM PUBLIC` nor per-function `revoke execute ... from public`. In PostgreSQL every function is granted EXECUTE to PUBLIC at creation, so `anon`/`authenticated` can invoke SECURITY DEFINER functions through PostgREST regardless of the selective grants. Functions that call `assert_admin()` in their body still fail closed (FORBIDDEN), so those are safe. The gap is `gate_metrics_evaluated()` (0011:362) — a SECURITY DEFINER function with NO internal authz check. It is intended for internal use only (called by the admin-gated `read_gate_metrics()` and the snapshot job, 0012:37), but the un-revoked PUBLIC default makes it directly callable, exposing program-health gate metrics (aggregate, not per-patron PII) to any caller. The selective-grant style elsewhere creates a false impression that un-granted functions are unreachable.
- **Fix**: Add a global `ALTER DEFAULT PRIVILEGES IN SCHEMA public REVOKE EXECUTE ON FUNCTIONS FROM PUBLIC;` early (e.g. in 0001/0002), then rely on the explicit per-function grants already present so only intended roles can call each RPC. For `gate_metrics_evaluated()` specifically, add `perform assert_admin();` (or revoke execute from anon/authenticated) so it fails closed even if called directly.

## [P3] perk_progress_count() takes arbitrary patron_id with no caller scoping, granted to authenticated
- **File**: supabase/migrations/0007_rpc_redeem.sql:28-56, grant at 236
- **What**: `perk_progress_count(p_patron_id, p_perk_id)` is SECURITY DEFINER with no `auth.uid()` / ownership check in its body and is `grant execute ... to authenticated` (line 236). It returns the stamp progress count for any `(patron_id, perk_id)` pair supplied by the caller. An authenticated caller who knows (or enumerates) patron UUIDs can read other patrons' progress counts toward any perk, a small cross-patron information leak relative to the Art. V boundary the rest of the model enforces. Impact is limited (a single integer; requires knowing target UUIDs) and the function is primarily an internal helper for `redeem_perk`/`checkin_progress` which do enforce ownership.
- **Fix**: Either revoke the direct `authenticated` grant (callers already reach it through the ownership-checked `redeem_perk`), or add an internal guard so a caller may only pass their own `current_patron_id()` unless they own the perk's business or are admin.
