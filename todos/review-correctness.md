# Correctness Review — 001-upper-delaware-passport

## [P2] redeem_perk has a read-modify-write race that can double-redeem one perk earning
- **File**: supabase/migrations/0007_rpc_redeem.sql:207-215
- **What**: `redeem_perk` reads `perk_progress_count(p_patron_ref, p_perk_id)`, checks `v_current < v_perk.visit_threshold`, then inserts a `perk_redemptions` row — with no row lock and no unique/exclusion constraint on `perk_redemptions`. Two redeem calls for the same (patron, perk) interleaving between the count read and the insert both observe `current >= threshold` and both insert a redemption. Because progress is "stamps strictly after the latest redemption", a single accumulation of stamps can be cashed in twice (or more), producing more redemptions than the patron earned. The same window exists if a redeem races a check-in.
- **Fix**: Serialize per patron+perk before the threshold check, e.g. `perform pg_advisory_xact_lock(hashtext(p_patron_ref::text || ':' || p_perk_id::text))` at the top of the function (after the perk lookup), OR take `select ... for update` on a representative row. Alternatively add a guard that the most-recent redemption is older than the newest counted stamp and re-read inside the same locked section. A DB-level partial unique constraint cannot express "one redemption per earning cycle", so an explicit lock is the correct fix.

## [P2] Stripe webhook dedup is check-then-insert without handling the unique-violation race
- **File**: supabase/functions/_shared/stripe-webhook-core.ts:114-125, 151-153, 260
- **What**: `handleStripeEvent` calls `alreadyProcessed` (SELECT on `processed_stripe_events`) and only at the end calls `markProcessed` (INSERT). `processed_stripe_events.event_id` is a PRIMARY KEY (0005_rpc_business.sql:12-14), but the two steps are not in one transaction and there is no row lock. When Stripe delivers the same event concurrently (its at-least-once delivery), both invocations pass the `alreadyProcessed=false` check, both apply the side effects (subscription update + business status change), and the second `markProcessed` INSERT throws a duplicate-key error that is not caught — surfacing as a 500 and a Stripe retry even though the work already completed. The side effects (e.g. `setBusinessStatus`) are applied twice; for `customer.subscription.updated` the second apply can reorder against a newer event and resurrect a stale status.
- **Fix**: Make markProcessed the gate: INSERT into `processed_stripe_events` FIRST with `on conflict (event_id) do nothing` and check whether a row was actually inserted (rowCount) before applying side effects; treat "no row inserted" as deduped. Wrap the whole handler in a DB transaction so the dedup insert and the side effects commit atomically.

## [P3] get_business_detail reports perk progress that ignores the redemption-reset rule
- **File**: supabase/migrations/0009_rpc_discovery.sql:151-166
- **What**: `get_business_detail` computes `v_my_count` as the patron's full season-lifetime trust-valid stamp count at the business and uses it directly as each perk's `current`. The load-bearing rule (documented in 0007_rpc_redeem.sql:6-13) is that perk progress counts only stamps created strictly after the patron's most-recent redemption of that perk. Every other surface (check-in confirmation, passport, redeem result, dashboard eligibility) uses `perk_progress_count`. As a result, after a patron redeems a perk, the business-detail screen keeps showing the pre-redemption (inflated) `current`/`ready` state instead of resetting to 0 — an inconsistent and incorrect progress value scoped to this one RPC.
- **Fix**: Replace `v_my_count` in the perks aggregate with `perk_progress_count(v_patron, p.id)` per perk (use a lateral/scalar subquery), matching 0007's rule. Keep `stamp_count` as the lifetime total if that field is intended to be lifetime, but `current`/`threshold`/`ready` must use the since-last-redemption count.

## [P3] get_business_regulars orders by visit count as TEXT, not numeric
- **File**: supabase/migrations/0013_rpc_regulars.sql:35
- **What**: The regulars list is built with `jsonb_agg(row order by row->>'visits' desc)`. `row->>'visits'` returns the count as TEXT, so the sort is lexicographic, not numeric. A patron with 9 visits sorts ahead of a patron with 10 visits (because text '9' > '10'), and 2 sorts ahead of 19. The "most frequent regulars first" ordering is therefore wrong whenever counts cross a digit-length boundary.
- **Fix**: Order by the numeric value, e.g. compute visits in the inner query and `order by (row->>'visits')::int desc`, or move the ORDER BY into the inner aggregate over the real integer column before building the jsonb.

## [P3 — REFUTED as written, logged for traceability] claim_passport merge constraint safety
- **File**: supabase/migrations/0006_rpc_checkin.sql:530-593
- **What**: The prior review flagged a P1 merge collision (duplicate stamps / unique constraints / concurrent claims). On trace, the duplicate-stamp handling is actually SAFE against the `stamps (patron_id, business_id, local_date)` unique key: conflicting loser stamps are voided and intentionally left on the loser row (their `not exists` re-point guard is false), while only non-conflicting loser stamps are re-pointed to the winner — so no unique violation and no row deletion (history preserved per Art. XIV). The `patrons.phone` unique key is not violated either: in the real flow the winner (phone holder, created by staff_check_in) holds the phone and the loser (auth-created via resolve_patron) has no phone. NOT a data-loss/constraint bug. See residual risks for the genuine (lower-probability) concurrency window.
